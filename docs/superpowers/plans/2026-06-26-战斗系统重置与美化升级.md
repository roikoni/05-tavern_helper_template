# 战斗系统重置与美化升级 实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 重置黑色修士角色卡的战斗系统——10 流派招牌机制差异化、`<combat>` 改用固定字段标签（结构自由）、战斗美化升级为正则兜底+脚本增强。

**Architecture:** 保留六维/D20/HP/MP/San 骨架不动；世界书新增流派机制总览与标签格式规范；正则只匹配 `<combat>` 边界包成 `data-combat` 容器（轻量静态美化兜底）；新增 `src/战斗美化/` 脚本扫描楼层 DOM，按 `<turn>/<school>/<action>/<response>/<roll>/<damage>/<status>` 标签解析，替换为带流派主题色与 CSS 动画的增强卡片。

**Tech Stack:** TypeScript、webpack（自动 glob `src/**/index.ts`）、酒馆助手接口（`tavern_events`、`displayed_message`）、jQuery（作用于酒馆父页面）、CSS 动画。无 Vue——战斗块渲染后数据不再变化，无响应式需求，模板字符串+CSS 动画更轻更稳（YAGNI）。

## Global Constraints

- 标签名固定为：`<turn>` `<school>` `<action>` `<response>` `<roll>` `<damage>` `<status>`，全小写英文，不得中文化。
- 流派清单固定为世界书 [修炼体系.yaml:23-33](src/黑色修士/世界书/修炼体系.yaml#L23) 的 10 个：剑修/法修/体修/丹修/阵修/符修/魂修/魔修/克苏鲁修/散修。不得臆造"血修/冰修/幻修/魅修"等。
- 战斗内状态（剑意/心魔/毒层等）只跨回合不跨战斗，**不写 MVU 长期变量**，仅出现在 `<school>` 标签文本里。
- 古神统一套"克苏鲁修"招牌机制做基底，专属神契技能叠加其上，不另设机制。
- 脚本以无沙盒 iframe 运行，`window.$` 作用于酒馆父页面（参考 [.cursor/rules/脚本.mdc:13](.cursor/rules/脚本.mdc#L13)）；加载用 `$(() => {})`，卸载用 `$(window).on('pagehide', ...)`，禁止 `DOMContentLoaded`/`unload`。
- 关键节点用 `console.info`，可恢复错误用 `console.warn/error`，顶层用 `errorCatched` 包裹。
- 本项目无单元测试框架（package.json 无 test 脚本，遵循现有模式），每个任务以 `pnpm build` 通过 + 手动酒馆验证作为完成标准。
- 构建产物路径：脚本项目 `src/战斗美化/` → `dist/战斗美化/index.js`，通过 `import 'https://.../dist/战斗美化/index.js'` 加载。

### 架构决策说明（偏离 spec 之处）

Spec 第六节写"挂载 Vue 组件"，本计划改用**模板字符串 + CSS 动画，不用 Vue**。理由：战斗块一旦由 LLM 输出并渲染，其数据不再变化，无响应式需求；聊天中会有大量战斗块，Vue 逐块挂载开销大、单 app + Teleport 散落 DOM 复杂度高。模板+CSS 动画更轻、更稳、维护更简（YAGNI）。若后续需要交互（折叠详情）可用原生 DOM 事件，无需 Vue。此偏离已在计划中标注，执行时若用户坚持 Vue 再调整。

```text
---
战斗格式:
  概述: |-
    文本中每有一次战斗动作，你都必须输出一个 `<combat>` 块记录本回合结算。你必须使用下方的固定字段标签，标签内的文字可以自由组织，但标签名不得更改、不得增删（非本回合适用的字段可省略）。

  字段标签:
    <turn>: 回合标题。必填。自由措辞，如"第3回合 — 奈亚的回合"。
    <school>: 流派与招牌状态。可选。格式为"流派名 · 状态描述"，如"剑修 · 剑意 3→5"、"克苏鲁修 · San 侵蚀"。无流派角色省略此标签。状态只跨回合不跨战斗，战斗结束清零。
    <action>: 主动方行动。必填。包含行动类型、技能/攻击名、消耗。
    <response>: 被动方应对。可选。无应对（如纯逃跑检定）时省略。
    <roll>: 检定过程与结果。必填。包含骰值、修正、DC、命中/擦伤/落空/暴击/大失败。
    <damage>: 伤害结算。可选。落空/无伤害时省略。须含骰子表达式、最终伤害数字、伤害类型。
    <status>: 状态变化。必填。列出双方 HP/MP/San 的变化与当前值，格式 HP:N/N | MP:N/N | San:N。

  格式模板: |-
    <combat>
    <turn>第N回合 — [主动方]的回合</turn>
    <school>[流派] · [招牌状态]</school>
    <action>[主动方]发动 [行动类型] · [技能/攻击名]（消耗: [MP N / 无消耗 / 自身扣除 San N]）</action>
    <response>[被动方]选择 [应对方式]（[效果简述]）</response>
    <roll>D20([骰值]) + [修正名]([修正值]) = [总值] vs [DC名]([DC值]) → [命中/擦伤/落空/暴击/大失败]</roll>
    <damage>[骰子表达式]([骰值]) + [修正] - [减免] = [最终伤害]点（[伤害类型]）</damage>
    <status>
    - [主动方] [HP/MP/San变化]，当前 HP: N/N | MP: N/N | San: N
    - [被动方] [HP/MP/San变化]，当前 HP: N/N | MP: N/N | San: N
    </status>
    </combat>

  示例: |-
    <combat>
    <turn>第3回合 — 奈亚的回合</turn>
    <school>克苏鲁修 · San 侵蚀</school>
    <action>奈亚发动 精神攻击 · 疯狂低语（消耗: MP 12 + 自身扣除 San 3）</action>
    <response>沈清辞选择 神识抵御（成功则精神伤害减半）</response>
    <roll>D20(17) + 神识修正(6) = 23 vs 神识抵御DC(18) → 命中</roll>
    <damage>D8(6) + 神识修正(6) - 神识减免(3) = 9点（精神/San）</damage>
    <status>
    - 奈亚 MP -12，San -3
    - 沈清辞 San -9，当前 HP: 160/160 | MP: 92/110 | San: 81
    </status>
    </combat>

  关键格式约束:
    - `<combat>` 与 `</combat>` 各占一行，标签内部不得有其他内容
    - 7 个字段标签名固定为 turn/school/action/response/roll/damage/status，全小写英文，不得更改
    - 标签内文字自由，但每个标签的语义信息必须完整
    - 字段顺序不强制，但建议按 turn → school → action → response → roll → damage → status 排列
    - 非本回合适用的字段可省略（如落空时省 <damage>，无应对时省 <response>，无流派时省 <school>）
    - 一次只结算一个回合，不要连续结算多个回合
    - 每个战斗叙事都可以生成一个回合
    - <status> 中的 HP/MP/San 格式必须为 HP: N/N | MP: N/N | San: N
    - <damage> 中最终伤害必须为数字，伤害类型用中文括号（）包裹
```

```text
  流派机制总览:
    概述: |-
      每个修炼流派拥有 1 个招牌机制与 1 个战斗内状态。状态只跨回合不跨战斗，战斗结束清零，记录在 <school> 标签中。凡人/无流派角色走纯通用规则，无招牌机制。古神统一套用克苏鲁修招牌机制（San 侵蚀）做基底，其专属神契技能叠加其上。

    剑修:
      状态: 剑意 N（初始 0）
      机制: 剑技或普通攻击命中时剑意 +1；剑意每层使剑技伤害 ×(1+0.2×剑意)。选择防御/闪避/反击/硬抗等应对时剑意清零，被眩晕/定身打断亦清零。
      节奏: 鼓励连续主动进攻，一旦转入应对即归零重蓄。

    法修:
      状态: 元素印（火/水/雷/风/土，各元素独立计数）
      机制: 法术攻击命中时累积对应元素印 1 枚。当场同时存在两种异元素印时触发"元素反应"追加效果并消耗这两枚印：火+风=扩散（伤害×1.5溅射）、水+雷=感电（目标下回合检定-3）、火+水=蒸发（本次伤害×2）、土+雷=超载（目标眩晕1回合）。
      节奏: 元素搭配策略，需规划攻击顺序凑反应。

    体修:
      状态: 架势（攻/守/崩，初始为守）
      机制: 主动方宣告行动前可切换架势。守=承受伤害-50%但自身伤害-30%；攻=自身伤害+30%但承受伤害+20%；崩=本回合硬抗（自动命中）且下回合必定暴击，但本回合承受伤害×1.5。
      节奏: 架势切换博弈，崩是赌命爆发。

    丹修:
      状态: 毒层 N（对敌）/ 药效（对己，临时强化层数）
      机制: 命中可附毒，毒层每回合自动结算 D4 HP 伤害，叠至上限(5)后不再叠加。丹修可消耗行动服丹：付 10% HP 换取下回合检定+3与伤害×1.5（药效1回合）。
      节奏: 正面伤害弱，靠毒伤 DOT 续航 + 服丹爆发窗口，控场拖延型。

    阵修:
      状态: 阵眼 N（初始 0）
      机制: 主动方可用行动布设 1 个阵眼（不造成伤害）。阵眼达 3 时触发阵法（消耗全部阵眼）：困敌阵=目标定身2回合、削弱阵=目标体魄/力道临时-5持续3回合、杀阵=对范围内全体造成 2D12 法术伤害。布阵期间只能硬抗或闪避。
      节奏: 正面弱且慢热，阵法一旦成型提供强力区域控制与群体减益。

    符修:
      状态: 符箓储备（种类×数量，战斗开始时按品级初始化）
      机制: 符修战斗开始拥有 3 道预制符（攻击符/防御符/控制符各1）。战斗中释放符为瞬发，不耗 MP、不占行动检定，但消耗符箓数量（每道符战斗结束不补）。可同回合叠多符连发。
      节奏: 正面持续输出弱（材料有限），但瞬发不耗 MP 提供灵活性与爆发窗口。

    魂修:
      状态: 魂噬 N（初始 0）
      机制: 神识攻击命中时魂噬 +1。魂噬达 3 时可发动"夺魂"（消耗3层）：目标下回合一回合行动受限（检定-5，不可选择反击）。魂修自身使用神识攻击有反噬风险：每次神识攻击检定自然1时自身扣 D4 San。
      节奏: 诡异控场，高风险高回报。

    魔修:
      状态: 心魔 N（初始 0）
      机制: 主动方可"燃血"：扣自身 10 HP 换取本次伤害×(1+0.3×心魔) 与吸血50%，心魔 +1。HP 低于 30% 时伤害额外×1.5。心魔达 5 时进入走火风险：每次行动检定自然1-5均视为大失败。
      节奏: 赌命型，HP 越低越强但高心魔不稳定。

    克苏鲁修:
      状态: 洗脑值/San（沿用基础数值，不另设计数）
      机制: 精神攻击累积洗脑值（每5点精神伤害目标洗脑+1），使用者每次精神技能自身扣 San（见克苏鲁精神系技能）。古神套用此基底，其专属神契技能（欲望权柄/欢宴幻梦/寂静共情/痴情之锁/深渊低语/千面之影/戏言之蝶）作为高阶精神技能叠加其上。
      节奏: 双刃侵蚀，伤敌亦伤己。

    散修:
      状态: 应变 N（初始 0）
      机制: 散修每回合可"借鉴"对手上一回合选择的应对方式一次（若对手上回合闪避，散修本回合闪避检定+2）。借鉴成功应变 +1，应变每层使所有检定+1（上限+3）。无专精上限，但没有流派专属伤害加成。
      节奏: 杂学应变，灵活但平庸。
```

```typescript
export interface CombatBlock {
  turn: string;
  school?: string;
  action: string;
  response?: string;
  roll: string;
  damage?: string;
  status: string;
  result?: string;
  damageValue?: number;
  schoolName?: string;
}

const TAGS = ['turn', 'school', 'action', 'response', 'roll', 'damage', 'status'] as const;

function extractTag(raw: string, tag: string): string | undefined {
  const re = new RegExp(`<${tag}>([\\s\\S]*?)</${tag}>`, 'i');
  const m = raw.match(re);
  return m ? m[1].trim() : undefined;
}

function extractResult(roll: string): string | undefined {
  const m = roll.match(/→\s*(命中|擦伤|落空|暴击|大失败)/);
  return m ? m[1] : undefined;
}

function extractDamageValue(damage: string): number | undefined {
  const m = damage.match(/=\s*(\d+)\s*点/);
  return m ? Number(m[1]) : undefined;
}

function extractSchoolName(school: string): string | undefined {
  const m = school.match(/^(剑修|法修|体修|丹修|阵修|符修|魂修|魔修|克苏鲁修|散修)/);
  return m ? m[1] : undefined;
}

export function parseCombat(raw: string): CombatBlock | null {
  const turn = extractTag(raw, 'turn');
  const action = extractTag(raw, 'action');
  const roll = extractTag(raw, 'roll');
  const status = extractTag(raw, 'status');
  if (!turn || !action || !roll || !status) return null;

  const school = extractTag(raw, 'school');
  const damage = extractTag(raw, 'damage');

  return {
    turn,
    school,
    action,
    response: extractTag(raw, 'response'),
    roll,
    damage,
    status,
    result: extractResult(roll),
    damageValue: damage ? extractDamageValue(damage) : undefined,
    schoolName: school ? extractSchoolName(school) : undefined,
  };
}
```

创建 `src/战斗美化/index.ts`：

```typescript
import { parseCombat } from './parser';
import { renderCombat, COMBAT_CSS } from './render';

const ENHANCED_CLASS = 'combat-enhanced';
const CSS_FLAG_ATTR = 'data-combat-css-injected';

/** 注入增强卡片 CSS 到酒馆父页面 head（只注一次） */
function injectCss(): void {
  if (document.querySelector(`style[${CSS_FLAG_ATTR}]`)) return;
  const style = document.createElement('style');
  style.setAttribute(CSS_FLAG_ATTR, '1');
  style.textContent = COMBAT_CSS;
  document.head.appendChild(style);
}

/** 扫描所有未增强的 data-combat 容器并替换为增强卡片 */
function enhanceAll(): void {
  const containers = document.querySelectorAll<HTMLDivElement>(`div[data-combat="1"]:not(.${ENHANCED_CLASS})`);
  if (containers.length === 0) return;

  containers.forEach(container => {
    const raw = container.innerHTML;
    const block = parseCombat(raw);
    if (!block) {
      // 解析失败：保留原兜底内容，仅标记已处理避免重复尝试
      container.classList.add(ENHANCED_CLASS);
      console.warn('[战斗美化] 解析失败，保留兜底显示:', raw.slice(0, 100));
      return;
    }
    container.innerHTML = renderCombat(block);
    container.classList.add(ENHANCED_CLASS);
  });
  console.info(`[战斗美化] 增强 ${containers.length} 个战斗块`);
}

$(() => {
  errorCatched(() => {
    injectCss();

    // 初始扫描（已存在的消息）
    enhanceAll();

    // 新消息到达后扫描
    eventOn(tavern_events.MESSAGE_RECEIVED, () => {
      // 延迟一帧，确保正则显示替换已完成
      requestAnimationFrame(() => enhanceAll());
    });

    // 重新生成消息后扫描
    eventOn(tavern_events.GENERATION_AFTER_COMMANDS, () => {
      requestAnimationFrame(() => enhanceAll());
    });

    console.info('[战斗美化脚本] 已加载');
  })();
});

$(window).on('pagehide', () => {
  console.info('[战斗美化脚本] 已卸载');
});
```

- `injectCss` 把 `COMBAT_CSS` 注入父页面 `<head>`（脚本 iframe 的 `document` 即酒馆父页面，因 `window.$` 指向父页；但 `document` 仍是 iframe 自己的——需确认）。见 Step 2 验证说明。

- [ ] **Step 2: 验证 document 指向（关键）**

脚本运行在 iframe 内，`document` 默认指 iframe 自己的 document，**不是**酒馆父页面的 document。但 [脚本.mdc:13](.cursor/rules/脚本.mdc#L13) 说明 `window.$` 被设为 `window.parent.$`。DOM 操作需用 jQuery（`$`）才能作用于父页面，原生 `document.querySelector` 只作用于 iframe。

因此 Step 1 的 `injectCss` 和 `enhanceAll` 若用原生 `document` API 会作用于错误的 document。**修正**：全部改用 jQuery `$`。

将 `index.ts` 中 `injectCss` 与 `enhanceAll` 改为：

```typescript
/** 注入增强卡片 CSS 到酒馆父页面 head（只注一次） */
function injectCss(): void {
  if ($(`style[${CSS_FLAG_ATTR}]`).length > 0) return;
  const $style = $(`<style ${CSS_FLAG_ATTR}="1"></style>`);
  $style.text(COMBAT_CSS);
  $('head').append($style);
}

/** 扫描所有未增强的 data-combat 容器并替换为增强卡片 */
function enhanceAll(): void {
  const $containers = $(`div[data-combat="1"]:not(.${ENHANCED_CLASS})`);
  if ($containers.length === 0) return;

  let count = 0;
  $containers.each((_, el) => {
    const $el = $(el);
    const raw = $el.html();
    const block = parseCombat(raw);
    if (!block) {
      $el.addClass(ENHANCED_CLASS);
      console.warn('[战斗美化] 解析失败，保留兜底显示:', raw.slice(0, 100));
      return;
    }
    $el.html(renderCombat(block));
    $el.addClass(ENHANCED_CLASS);
    count++;
  });
  if (count > 0) console.info(`[战斗美化] 增强 ${count} 个战斗块`);
}
```

注意：`$` 这里是父页面的 jQuery，`$('head')`、`$('div[data-combat="1"]')` 作用于酒馆网页。`parseCombat` 接收的 `raw` 是 `$el.html()`——但 `innerHTML` 会把 `<turn>...</turn>` 序列化回原始标签文本，解析器能正确匹配。✓
